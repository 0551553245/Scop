This is Scop — a restaurant operations SaaS for Saudi Arabia. Three panels: Branch Manager at /branch-manager/, Owner at /owner/, Super Admin at /admin/. Tech stack: React plus Vite, Supabase PostgreSQL, Node.js Express. Supabase URL: https://bjjpcawqkwufttjabuol.supabase.co. Three separate Supabase clients — supabaseBranchManager with storageKey scop-bm-session, supabaseOwner with storageKey scop-owner-session, supabaseAdmin with storageKey scop-admin-session. Never share one client across panels. Three auth contexts — BranchManagerAuthContext, OwnerAuthContext, AdminAuthContext — never use a single shared AuthContext. Manager accounts are created by owners directly — no self-registration, no invite codes. Status colors: green #2D7A4F completed, yellow #F5A623 pending, red #E24B4A missed, blue #378ADD scheduled. Database tables: users has id email name name_ar phone role branch_id is_active created_at. branches has id name name_ar city city_ar owner_id manager_id is_active created_at. tasks has id name name_ar description description_ar frequency branch_id is_active requires_photo requires_note requires_value value_unit value_min value_max category created_by created_at. task_submissions has id task_id branch_id submitted_by status note photo_url value_entered submitted_at — NO submission_date column. food_safety_standards has id name name_ar description branch_id min_temp max_temp standard_type unit is_active created_by created_at — standard_type is temperature, weight, or compliance, unit is °C, kg, g, or null for compliance. food_safety_submissions has id standard_id branch_id submitted_by result actual_value corrective_note submitted_at — uses result not status, NO submission_date column. Critical rules: always use submitted_at range filtering never submission_date. Always use .or(branch_id.eq.X,branch_id.is.null) for task queries. Always include is_active true on task inserts. food_safety_submissions uses result not status.
